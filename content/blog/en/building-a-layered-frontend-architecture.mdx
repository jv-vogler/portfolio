## The Problem with Flat Structures

Most frontend projects start with a flat structure: components, hooks, utils, and maybe a services folder. This works fine for small apps, but as a project grows, it becomes harder to reason about **dependency flow** and **separation of concerns**.

When your utility file imports from your component folder, and your component imports from your API layer, you've created a tangled dependency graph that's hard to maintain or test.

### The Layered Architecture

I adopted a **layered frontend architecture** with four distinct layers, each with strict dependency rules:

```
core/ → Pure domain types & logic (zero framework imports)
lib/  → Generic utilities (no domain knowledge)
app/  → Server Actions, data access, I/O
ui/   → React components, hooks, pages
```

### The Golden Rule

**Inner layers never import from outer layers.** `core/` knows nothing about React. `lib/` knows nothing about your domain. This makes the inner layers trivially testable and portable.

### Core Layer

The `core/` layer contains pure TypeScript and Zod schemas — no React, no Next.js, no framework code at all:

```typescript
// core/contact.ts
export namespace Contact {
  export const formSchema = z.object({
    name: z.string().min(2),
    email: z.string().email(),
    message: z.string().min(20),
  })

  export type FormValues = z.infer<typeof formSchema>
}
```

This schema is used by both the UI (React Hook Form + Zod resolver) and the server action (validation before sending email). One source of truth, zero duplication.

### Lib Layer

The `lib/` layer provides generic, domain-agnostic utilities:

```typescript
// lib/date.ts
export function formatDate(dateString: string, locale: string): string {
  return new Intl.DateTimeFormat(locale, {
    year: 'numeric',
    month: 'long',
    day: 'numeric',
  }).format(new Date(dateString))
}
```

### Benefits

1. **Testability** — Core and lib layers can be unit tested without any framework setup.
2. **Portability** — If you ever switch from Next.js to Remix or Astro, your core layer comes with you unchanged.
3. **Clarity** — When you see an import from `@/core/contact`, you know it's pure domain logic. When you see `@/app/actions/contact`, you know it touches the server.

### Conclusion

A layered architecture adds a small amount of structure upfront but pays dividends in maintainability. For a portfolio site, it might seem like overkill — but it's also the perfect sandbox to practice patterns you'll use in larger applications.
