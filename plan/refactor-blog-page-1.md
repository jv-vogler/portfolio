# Introduction

![Status: In Progress](https://img.shields.io/badge/status-In%20Progress-yellow)

A comprehensive refactor of the blog subsystem (`/blog` list page and `/blog/[slug]` post page) to transform it from a minimal MVP into a polished, feature-rich developer blog. The refactor adds discovery features (tag filtering, command palette, related posts), content enrichment (reading time, TOC, cover images), UX polish (progress bar, back-to-top, animated transitions, share buttons), and distribution (RSS feed). All features are URL-synced where applicable, respect the existing i18n setup (en/pt), and follow the established patterns (Payload collections, `Blog` namespace in `src/core/blog.ts`, shadcn/ui components, `next-intl`, framer-motion).

## 1. Requirements & Constraints

- **REQ-001**: Tag filtering must be URL-synced via search params (`?tag=react`) so filtered views are shareable and bookmarkable. Multiple tags use `?tag=react&tag=nextjs` with AND logic.
- **REQ-002**: Reading time must be computed from the Lexical `SerializedEditorState` content by recursively extracting text nodes and counting words. Display as "X min read" with locale-aware formatting.
- **REQ-003**: Table of contents (TOC) must be auto-generated from heading nodes (h2, h3) in the Lexical content. Headings must have stable `id` attributes for anchor linking. TOC must highlight the current section via scroll-spy.
- **REQ-004**: Related posts must be determined by shared-tag overlap, showing up to 3 posts ranked by number of matching tags. Falls back to most-recent posts if no tags match.
- **REQ-005**: Reading progress bar must be a thin, fixed-position horizontal bar at the top of the viewport on individual post pages. Uses `scroll` event with `requestAnimationFrame` throttling.
- **REQ-006**: RSS feed must be served at `/feed.xml` as a valid Atom/RSS 2.0 feed. Must include all published posts with title, description, date, tags, and a link to the full post. Must include both locales.
- **REQ-007**: Code blocks rendered inside the Lexical rich text must have a "Copy" button that copies the code content to the clipboard and shows a brief confirmation.
- **REQ-008**: Spotlight/featured post must use a `featured` checkbox field on the Posts collection (same pattern as Projects). The featured post renders at the top of the list with a distinct, larger card layout.
- **REQ-009**: Command palette must use `cmdk` via the shadcn/ui `<Command>` component. Triggered by `⌘K` / `Ctrl+K` or a button in the header. Must search all navigation links, blog posts, and social links. Must be i18n-aware.
- **REQ-010**: Anchor link headings: all h2/h3 in rendered blog posts must have a hoverable `#` link icon that copies a deep-link URL to that heading.
- **REQ-011**: Updated badge: if `updatedAt` is >24h after `publishedDate`, show "Updated [date]" badge near the post date.
- **REQ-012**: Animated transitions: use `framer-motion` `AnimatePresence` for page transitions between the blog list and individual posts. Respect `prefers-reduced-motion`.
- **REQ-013**: Back-to-top: a floating button that appears after scrolling 300px down on individual post pages. Smooth-scrolls to top on click.
- **REQ-014**: Share buttons: share to X/Twitter, LinkedIn, and a "Copy link" button on individual post pages. Use Web Share API on mobile where available.
- **REQ-015**: Post cover images: add a `coverImage` relationship field (to `media` collection) on the Posts collection. Display as a hero banner on the post page and as a card thumbnail in the blog list.
- **REQ-016**: All user-facing text must be added to both `src/messages/en.json` and `src/messages/pt.json` with appropriate translation keys under the `blog` namespace (or a new namespace where appropriate, e.g. `commandPalette`).
- **SEC-001**: Share buttons must not leak any user data. Copy-to-clipboard must fail gracefully if the Clipboard API is unavailable (fallback to `document.execCommand`).
- **CON-001**: The project uses Next.js 16 with App Router, `next-intl` v4, Payload 3.78, Tailwind CSS v4, `framer-motion` v12, `shadcn/ui` (radix + CVA), and `pnpm` as the package manager.
- **CON-002**: No changes to the Payload admin panel layout are required—only collection schema changes.
- **CON-003**: The `coverImage` field must be optional (backward-compatible with existing posts that have no image).
- **CON-004**: All new client components must use the `"use client"` directive. Server components are the default.
- **CON-005**: Payload database migrations must be generated after schema changes (`pnpm payload migrate:create`).
- **GUD-001**: Follow the existing pattern of core domain types in `src/core/blog.ts` (the `Blog` namespace), server actions in `src/app/actions/blog.ts`, and UI components in `src/ui/blog/components/`.
- **GUD-002**: Use the established `MotionSection`, `fadeInUp`, `staggerContainer` animation variants from `src/ui/lib/motion.tsx` where possible. Create new variants only when needed.
- **GUD-003**: Use `lucide-react` icons consistently (already a project dependency).
- **GUD-004**: Use the `storage` helper from `src/lib/storage.ts` for any client-side preference persistence (e.g. recently searched terms in command palette).
- **PAT-001**: Follow the `featured` checkbox pattern from `src/collections/Projects.ts` for the spotlight post field.
- **PAT-002**: Follow the `BlogPostingJsonLd` pattern from `src/ui/lib/jsonLd.tsx` when enriching structured data with new fields (cover image, dateModified).

## 3. Alternatives

- **ALT-001**: **Client-side tag filtering** — Filter entirely on the client after fetching all posts. Rejected because URL-synced server-side filtering provides shareable URLs and better SEO. The `searchParams` approach also works with ISR caching.
- **ALT-002**: **kbar instead of cmdk** — `kbar` is another popular command palette library. Rejected because shadcn/ui has built-in `cmdk` integration via the `<Command>` component, which matches the project's existing UI pattern and avoids adding a separate design system dependency.
- **ALT-003**: **View Transitions API** for page transitions — The native View Transitions API is modern and performant, but browser support is limited (no Firefox as of March 2026) and the existing framer-motion dependency already provides this capability with wider compatibility.
- **ALT-004**: **Separate reading-time npm package** — Packages like `reading-time` exist, but they work with plain text/markdown. Since content is Lexical `SerializedEditorState`, a custom 15-line recursive text extractor is simpler and has zero added dependencies.
- **ALT-005**: **Dedicated Payload `tags` collection** (relational) instead of inline array — A dedicated collection allows tag management via admin UI and enforces consistency. Deferred for now since the inline tags array is already in production; can be migrated later without frontend changes.
- **ALT-006**: **Use `next/og` for dynamic OG images with cover images** — Would be a nice addition but is out of scope for this refactor. Can be layered on later.

## 4. Dependencies

- **DEP-001**: `cmdk` — Installed automatically by `shadcn add command`. Required for the command palette. (~8KB gzipped)
- **DEP-002**: `@radix-ui/react-dialog` — Installed automatically by `shadcn add dialog`. Required for the command palette dialog wrapper. Already partially available via radix-ui dependency.
- **DEP-003**: No other new npm dependencies required. All other features use existing deps: `framer-motion` (animations), `lucide-react` (icons), `next/image` (cover images), `@payloadcms/richtext-lexical` (custom converters), `simple-icons` (social share icons).

## 7. Risks & Assumptions

- **RISK-001**: **Lexical node structure changes** — The reading time and heading extraction functions depend on the Lexical serialization format. If `@payloadcms/richtext-lexical` changes the node structure in a future update, these functions may break. Mitigated by pinning Payload/Lexical versions and testing with actual content.
- **RISK-002**: **Custom RichText converters compatibility** — The `<RichText>` component from `@payloadcms/richtext-lexical/react` supports a `converters` prop, but the exact API for overriding heading and code block renderers may require consulting Payload docs. If the converter API doesn't support the desired customization, a post-processing approach (wrapping the rendered HTML) may be needed.
- **RISK-003**: **Performance of command palette blog post loading** — Loading all blog post titles on mount in the `CommandPaletteProvider` adds a server action call per page load. Acceptable at small scale (~tens of posts). If post count grows to hundreds, this should be refactored to on-demand search.
- **RISK-004**: **AnimatePresence with App Router** — App Router doesn't natively support exit animations between route segments. The `AnimatePresence` approach at the blog layout level may not reliably trigger exit animations. May need to fall back to simple enter-only animations or use the `next-view-transitions` package.
- **ASSUMPTION-001**: The `@payloadcms/richtext-lexical/react` `<RichText>` component supports a `converters` or `renderers` prop for customizing how specific node types are rendered. This needs to be verified against the installed version (3.78.0).
- **ASSUMPTION-002**: The existing `media` collection image sizes (`thumbnail: 400w`, `card: 768w`, `hero: 1400w`) are sufficient for cover image rendering at all breakpoints.
- **ASSUMPTION-003**: The current number of blog posts is small enough that fetching all posts for tag extraction, related posts, and command palette indexing is performant without pagination.
- **ASSUMPTION-004**: The Payload Lexical editor's heading nodes use `type: "heading"` with a `tag` property of `"h1"` through `"h6"`. This follows the standard Lexical heading node format.
